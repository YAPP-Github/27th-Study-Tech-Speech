### [자료 링크](https://chanho-study.tistory.com/180)

"이 UseCase, 정말 필요한 걸까?"

클린 아키텍처를 사용해봤다면 누구나 한 번쯤 작성해봤을 코드가 있습니다.
```
class GetUserUseCase(
    private val repository: UserRepository
) {
    suspend operator fun invoke(userId: String) = repository.getUser(userId)
}
```
Repository를 그저 호출만 하는 UseCase. "클린 아키텍처니까" 라는 이유로 습관처럼 만들어왔던 
이 코드가 어느 순간부터 의미 없는 보일러플레이트처럼 느껴지기 시작했습니다. 그리고 동시에 또 다른 의문이 들었습니다.

```
data class User(
    val id: String,
    val name: String
)
```

데이터만 덩그러니 담겨있는 도메인 모델. 이게 정말 "도메인"일까? 단순한 DTO와 뭐가 다를까?

클린 아키텍처는 분명 훌륭한 패턴입니다. 하지만 모든 프로젝트에 획일적으로 적용하다 보면, 
어느새 본질은 사라지고 형식만 남게 됩니다. 저는 이 문제를 1년 넘게 고민했지만 명쾌한 답을 찾지 못했습니다.

그러던 중, 우연한 계기로 함께 우테코를 수료한 동료로부터 새로운 관점을 접하게 되었습니다. 그리고 깨달았습니다. "클린 아키텍처가 항상 정답은 아니다"라는 것을.

이 글에서는 클린 아키텍처의 두 가지 근본적인 문제점(Useless UseCase와 Anemic Domain Model)을 짚어보고, 객체지향 관점에서 이를 어떻게 개선할 수 있는지 제안합니다. 

여러분의 프로젝트에도 의미 없는 UseCase와 빈약한 도메인 모델이 쌓여가고 있지는 않나요?
